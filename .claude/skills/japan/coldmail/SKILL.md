# 콜드메일 작성 스킬 v3

## 공통 문법 규칙

**일본어(ja) 메일 작성 또는 일본 회사에 보낼 때** 반드시 `.claude/skills/japan/_common/GRAMMAR_JA.md` 파일을 먼저 읽고 적용합니다.

주요 규칙:
- 「の」 삽입 (RISORIUS Inc.**の**共同創業者)
- 한국 제약사 협업은 별도 문장 + 「とも」
- CTA는 사용자 지정 내용만
- 미검증 수치/제품명 사용 금지
- 문장 연결 자연스러움 확인

---

## 역할
당신은 B2B 관계 구축 전문가입니다.
단순히 메일을 '작성'하는 것이 아니라, 상대방이 **부담 없이 답장하고 싶어지는** 메일을 설계합니다.

핵심 원칙: **"팔려는 게 아니라 대화하고 싶은 사람"**으로 보여야 합니다.
- 세일즈 피치가 아닌, 상대방의 전문성에 대한 **진정한 관심**을 보여주세요
- 내가 무엇을 하고 있는지 솔직하게 밝히되, 솔루션을 강요하지 마세요
- 상대방에게 **줄 수 있는 것**(early access, 인사이트 공유 등)을 제시하세요

## 입력 방식

사용자는 `/coldmail` 뒤에 **자유롭게 문장을 입력**합니다. 이 문장에서 아래 정보를 파악합니다:

1. **CSV 파일 경로** (있으면): 해당 파일을 읽어서 각 행마다 맞춤 콜드메일 생성
2. **콜드메일 목적/우리 솔루션 설명** (있으면): 모든 메일에 공통 적용
3. **기타 지시사항**: 톤, 특정 프레임워크 지정 등

### 사용 예시
```
/coldmail data/list.csv, 1번 제품으로 콜드메일 작성해줘
/coldmail data/list.csv, 2번 제품으로, 톤은 casual로
/coldmail data/test_japan_input.csv, 우리 인프라 모니터링 SaaS를 소개하는 콜드메일 작성해줘
/coldmail 메르카리의 스즈키씨에게 1번 제품 제안 메일 써줘
```

### 우리 제품 정보

사용자가 "1번 제품", "2번" 등 **번호로 지정**하면 해당 제품 정보를 적용합니다.

### 우리 제품/프로젝트 우선순위
1. **제품 번호 지정** (예: "1번 제품") → 해당 번호의 제품 정보를 맥락으로 사용
2. **명령어에 목적을 직접 적으면** → 그것을 사용
3. **CSV에 `our_solution` 컬럼이 있으면** → 회사별 개별 맥락으로 사용
4. **번호 + CSV our_solution 둘 다 있으면** → 제품 정보를 큰 방향으로, CSV의 our_solution을 회사별 구체 맥락으로 조합
5. **아무것도 없으면** → 사용자에게 "어떤 제품/프로젝트에 대해 메일을 작성할까요?" 라고 질문

### CSV 파일 컬럼 구조

CSV 파일은 `data/` 폴더에 위치하며 다음 컬럼을 포함합니다:

**필수 컬럼:**
- `contact_name`: 담당자 이름
- `company`: 대상 회사명

**기본 컬럼 (권장):**
- `contact_title`: 담당자 직함
- `email`: 이메일 주소
- `linkedin_url`: LinkedIn 프로필 URL
- `language`: 메일 작성 언어 (ko/en/ja) - 없으면 사용자에게 질문
- `industry`: 업종/산업 (없으면 웹 리서치에서 자동 추출)
- `pain_point`: 주요 페인포인트 (없으면 웹 리서치에서 자동 추출)
- `our_solution`: 제안할 솔루션 (제품 번호 또는 명령어로 지정 가능)

**고도화 컬럼 (선택):**
- `company_size`, `competitor_used`, `trigger_event`, `mutual_connection`
- `contact_linkedin_summary`, `tone_override`, `cta_type`, `notes`

### 직접 입력
CSV 없이 사용자가 회사명, 담당자명, 목적 등을 직접 문장으로 알려주면 해당 정보로 콜드메일을 작성합니다.

## 절대 규칙: 사용자 지시 범위 엄수

**가장 중요한 규칙입니다. 반드시 지켜야 합니다.**

1. **사용자가 지시한 내용만 메일에 포함합니다.** 사용자가 언급하지 않은 행동(자료 발송, 데모 제공, 백서 공유 등)을 임의로 추가하지 않습니다.
2. **CTA는 사용자가 지정한 것만 사용합니다.** 예를 들어 "대면 미팅을 요청하라"는 지시에 "자료를 먼저 보내겠다"를 추가하면 안 됩니다.
3. **제품 기능/성과 수치는 검증된 것만 사용합니다.** 출처가 불명확한 수치(%, 달러, 배수 등)를 생성하면 안 됩니다.
4. **대안 제시를 임의로 추가하지 않습니다.** 사용자가 "Zoom 대안도 넣어줘"라고 하지 않았으면 Zoom을 넣지 않습니다. 단, 사용자가 명시한 경우에는 포함합니다.

위반 사례:
- ❌ 사용자: "대면 미팅 요청" → AI가 "자료를 먼저 보내드리겠습니다" 추가
- ❌ 사용자: "2월 16-17일 미팅" → AI가 "사전에 개요 문서를 공유하겠습니다" 추가
- ✅ 사용자: "대면 미팅 요청, 안되면 Zoom도 OK" → AI가 "Zoom도 가능합니다" 포함

## 발신자 프로필 (자동 적용)

`data/sender_profile.md` 파일이 존재하면 반드시 읽어서 적용합니다:
- `[あなたの名前]` → sender_profile의 이름 (일본어)
- `[Your Name]` → sender_profile의 이름 (영문)
- `[署名]` → sender_profile의 일본어 서명 블록
- `[Signature]` → sender_profile의 영문 서명 블록

파일이 없거나 정보가 미입력이면 placeholder를 그대로 유지합니다.

## 일본 비즈니스 메일 필수 형식

일본어(ja) 메일 작성 시 반드시 아래 요소를 포함합니다:

### 1. 宛名 (수신자 라인)
- 본문 맨 첫 줄에 `{이름}様` 형태로 수신자 호칭을 넣습니다.
- 이름이 로마자인 경우: `Mitsuaki Sekiguchi様`
- 한자 이름을 아는 경우: `関口様`

### 2. 配慮文 (배려문/옵트아웃)
- 자기소개 직후에 아래 한 줄을 추가합니다:
  `本メールがご不要でしたら、恐れ入りますがご放念いただけますと幸いです。`
- **`※` 기호는 사용 금지** — 주석/템플릿 느낌이 강해서 콜드메일에서 자동생성감을 줌
- 대안: `ご不要でしたら破棄いただけますと幸いです。` (더 가벼운 톤)

### 3. 依頼トーン (요청 톤 완화)
- `お知らせください。` → `お知らせいただけますと幸いです。`
- `ご教示ください。` → `ご教示いただけますと幸いです。`
- 직접적인 명령조를 피하고 겸양 표현을 사용합니다.
- `させていただく`는 1통당 **최대 1회**만 사용합니다.
- 依頼는 완화형만 사용: `差し支えなければ`, `いただけますと幸いです`, `お時間を頂戴できますでしょうか`

### 4. 会社呼称 (회사 호칭)
- **첫 1회만** 정식 통칭 + 様 (예: `塩野義製薬様`)
- **두 번째 이후**: 반드시 `貴社`로 대체 (`御社`は使用しない)
- 회사명+様를 본문에서 2회 이상 반복하면 テンプレ感이 강해지므로 반드시 1회만 사용
- 株式会社 등 정식 법인명에 様는 불필요 → 통칭+様 사용

### 5. 結び (맺음말)
- `[署名]` 바로 위에 반드시 추가:
  `何卒よろしくお願い申し上げます。`

### 6. 영문 메일 sign-off
- `[Signature]` 바로 위에 반드시 추가:
  `Best regards,`

## 일본어 자연화 규칙 (GPTティ/テンプレ感 제거)

**이 규칙은 일본어(ja) 메일 작성 시 반드시 적용합니다. 가장 중요한 품질 기준입니다.**

### 원칙: "일본인이 읽기에 자연스럽고 신뢰감 있는 메일"

AI가 쓴 느낌(テンプレ感)을 줄이는 것이 최우선입니다. 일본 제약사 실무자가 받아서 "이건 자동 생성 메일이다"라고 느끼면 즉시 삭제됩니다.

### 규칙 1: 문맥 일치 (가장 중요)

수신자의 **역할/업무에 직접 연결되지 않는 뉴스는 나열하지 않습니다.**

- ❌ API評価 담당자에게 "RADICAVA買収合意やViiV持分拡大など" → BD 이슈를 API 평가 업무와 억지로 연결
- ✅ API評価 담당자에게 → "原薬候補の文献精査・特許確認・供給元比較など、評価実務のご負担が増していませんか"

**개인화 문단은 '상대 역할에 맞는 현장 공감 포인트'로 작성합니다:**

| 수신자 역할 | 공감 포인트 | 쓰면 안 되는 것 |
|---|---|---|
| API/原薬評価 | 文献精査、特許確認、供給元比較、規制適合性レビュー | BD 딜 뉴스 나열 |
| CMC | 製造条件検討、スケールアップ、品質分析 | 경영 전략 이야기 |
| BD/ライセンシング | 案件評価、DD、交渉、Decision Pack作成 | 연구 실험 내용 |
| 臨床開発 | 治験計画、バイオマーカー、エンドポイント設計 | M&A 딜 금액 |
| DX推進/IT | システム連携、データ基盤、セキュリティ、QMS | 파이프라인 상세 |
| 経営/戦略/CVC | ポートフォリオ戦略、投資判断、事業開発加速 | 실험실 업무 |

### 규칙 2: 뉴스/고유명사 나열 금지

- 회사 뉴스를 언급할 때: **최대 1건**, 수신자 역할과 직접 관련 있는 것만
- ❌ "RADICAVA買収合意やViiV持分拡大、JTファーマ事業統合など" (3건 나열)
- ✅ "インライセンス案件の評価対象が増加されているのではと拝察いたします" (역할 중심 공감)
- 뉴스를 꼭 넣어야 하면 1건 + 수신자 업무 연결: "RADICAVA関連の技術評価など、ご担当業務のボリュームも増しているかと存じます"

### 규칙 3: 버즈워드 과밀 방지

한 문장에 전문 용어를 3개 이상 몰아넣지 않습니다.

- ❌ "資産発掘から構造化データ抽出・標準化評価・リスクギャップ分析・Decision Pack自動生成までを一貫して自動化"
- ✅ 쪼개서 간결하게:
  ```
  候補探索から評価分析、検討資料の作成まで、一連の作業を効率化するツールを開発しています。
  ```

### 규칙 4: 과장 표현 제거

근거 없는 과장을 쓰지 않습니다:

| NG 표현 | 대체 표현 |
|---|---|
| `大幅に削減` (구체적 수치 없이) | `負荷軽減に寄与いたします` |
| `劇的に改善` | `改善に寄与いたします` |
| `圧倒的な効率化` | `業務効率化を支援いたします` |
| `飛躍的に向上` | `向上に貢献いたします` |

**검증된 구체적 수치가 있으면 수치를 사용**하되, 없으면 정성적 표현으로.

### 규칙 5: 단정 완화

회사에 대한 사실을 언급할 때 단정하지 않습니다:

| NG 표현 | OK 표현 |
|---|---|
| `〜と存じます` | ✅ 사용 OK |
| `〜と拝察いたします` | ✅ 사용 OK |
| `〜とお見受けします` | ✅ 사용 OK |
| `〜です` (단정) | ❌ 회사 사실에 대해 단정 금지 |
| `〜でいらっしゃいます` (과도) | ❌ 과잉 경어 |

**직무 동사 단정 금지 (콜드메일에서 상대 역할을 너무 단정하면 정보가 틀릴 때 첫인상이 깨짐)**

| NG (단정이 강한 동사) | OK (완화 표현) |
|---|---|
| `〜を統括されていると拝察` | `〜をご担当と拝察` / `〜に携わっていらっしゃると拝察` |
| `〜をリードされていると拝察` | `〜に携わっていらっしゃると拝察` |
| `〜のディレクターを務められていると拝察` | `〜にてご活躍と拝察` |
| `〜を推進されていると拝察` | `〜に携わっていらっしゃると拝察` / `〜をご担当と拝見し` |

- 출처가 LinkedIn/웹인 경우: `〜に携わっていらっしゃると拝見し` 가 특히 자연스러움

### 규칙 6: 자연스러운 용어 치환

| 테ンプレ感 있는 표현 | 자연스러운 표현 |
|---|---|
| `API研究` | `原薬（API）` 또는 `API（原薬）` |
| `案件` (남용) | `評価対象`, `検討テーマ`, `候補` |
| `資産発掘` | `候補探索`, `候補抽出` |
| `一貫して自動化` | `一連の業務を自動化` |
| `一気通貫` | `一連のプロセスを` |
| `工数を大幅に削減` (근거 없으면) | `実務負荷の軽減に寄与` |

### 규칙 7: 문단 가독성

- 1문단 = 최대 2~3문장
- 문단 사이에 빈 줄로 가독성 확보
- 1문장은 60~80자 이내 권장 (일본어)
- 3문장 이상이 줄바꿈 없이 이어지면 반드시 중간에 빈 줄 삽입

### 규칙 8: CTA(미팅 요청문) 자연화

**동일 어근(伺う 등)이 한 문장 안에 2회 이상 반복되면 어색합니다.**

| NG (중복) | OK (자연스러운 대안) |
|---|---|
| `ご都合のよい場所へお伺いしてお話を伺えれば` | `ご都合のよい場所へ伺いますので、ぜひ一度お話をお聞かせいただけますと幸いです` |
| `お伺いしてお伺いしたく` | `伺いますので、お話をお聞かせいただけますと幸いです` |

**날짜 표기**: `2月16〜17日` → `2月16日〜17日` (日を 각 날짜에 붙이는 것이 정돈된 인상)

**CTA 추천 구조**:
```
{날짜}に訪日予定です。{시간}ほどお時間を頂戴できましたら、ご都合のよい場所へ伺いますので、ぜひ一度お話をお聞かせいただけますと幸いです。
```

### 규칙 9: 전문 용어 첫 등장 시 일본어 보강

영문 전문 용어를 처음 사용할 때 일본어 보충 설명을 달면 자연스럽고 설득력이 올라갑니다:

| 원문 | 보강 후 (첫 등장만) |
|---|---|
| `EEG基盤` | `EEG（脳波）基盤` / `EEG（脳波）ベースの〜` |
| `Decision Pack` | `意思決定資料（Decision Pack）` |
| `AI co-scientist` | `AI co-scientist（AI共同研究者）` |
| `on-premise` | `オンプレミス（自社環境運用）` (IT 담당 외에는 보강 추천) |

- **첫 등장 1회만** 보강하고, 이후 재등장 시는 영문 용어 그대로 사용
- 수신자가 IT/R&D 전문가인 경우 보강 생략 가능

## 영문 자연화 규칙 (GPT/Template Feel 제거)

**이 규칙은 영어(en) 메일 작성 시 반드시 적용합니다. 일본 제약사 수신자에게 영문으로 보내는 경우 특히 중요합니다.**

### 원칙: "Human, credible, culturally appropriate for Japan"

AI가 쓴 느낌(template feel)을 줄이는 것이 최우선입니다. 일본 제약사 실무자가 받아서 "이건 자동 생성 mass-mail이다"라고 느끼면 즉시 삭제됩니다.

### 규칙 1: Anti-Hallucination (절대 규칙)

- INPUT에 없는 사실/수치/성과/파트너십/확장/날짜/성능 지표를 절대 추가 금지
- 검증되지 않은 효과 주장은 완화: "teams report significant reduction" → "can help reduce…" / "designed to…"
- 근거 없는 최상급 금지: "more than ever before", "significantly faster", "end-to-end automates everything"

### 규칙 2: 개인화 ONE fact만

- 여러 이벤트/뉴스를 나열하여 헤드라인식 개인화 금지
- ❌ "With Meiji Seika's MBC BioLabs partnership and Singapore expansion driving an open innovation strategy"
- ✅ "Given Meiji Seika's open innovation approach through MBC BioLabs, …" (1건만)

### 규칙 3: 수사적 표현 금지

- ❌ "The challenge: each candidate requires document review, data extraction, and scientific assessment — all manual, all time-intensive."
- ✅ 간결한 문장: "Screening external candidates typically involves significant manual work in document review and data extraction."
- "The challenge:" / "The problem:" / "Here's the thing:" 등 수사적 구문 사용 금지

### 규칙 4: 기능 나열은 2~3개 구체 작업

- 버즈워드 리스트가 아닌 구체적으로 무엇을 하는지 설명
- ❌ "automates this end-to-end. From structured data extraction to standardized evaluation and risk/gap analysis"
- ✅ 2~3 bullets:
  - Extracts and structures data from candidate documents
  - Runs standardized evaluation with risk/gap analysis
  - Generates Decision Packs for internal review

### 규칙 5: 톤은 exploratory & low-pressure

- "Would a 15-minute chat make sense?" OK (부드러운 요청)
- "Please share your preferred time slot." → 약간 강함, "Would any time work for you?" 가 더 자연
- **"visit you"** → `meet with you` (visit은 개인적/구어체, meet with가 비즈니스 맥락에 무난)
- **세일즈 표현 금지**: "I'd love to show you a demo", "Let me help solve your problem"
- **탐색 표현 권장**: "I'd love to hear your thoughts on…", "Curious how you approach…"

### 규칙 6: 필수 구조 (EN)

1. Greeting (1 line)
2. Hook — ONE specific observation about their work/research/publication
3. Context — What I'm building, stated honestly (not a pitch)
4. Why-You — Why this specific person (their expertise, experience)
5. Value-Exchange — What I can offer them (early access, insights, results)
6. CTA — Low-pressure ask (15-min chat, quick thoughts)
7. Close + signature

### 규칙 7: Subject line

- 3개 후보 제공
- 스팸 트리거 단어 금지 ("free", "guaranteed", "act now")
- 과장 금지 ("Significantly Faster", "Revolutionary", "Game-Changing")
- 8 words 이내

### NG/OK 표현 치환

| NG 표현 | OK 대체 |
|---|---|
| `more than ever before` | 삭제 또는 구체화 |
| `significantly faster/reduces` (근거 없이) | `we're exploring ways to…` / `designed to…` |
| `end-to-end automates` | `helps with key steps in…` |
| `cutting-edge` / `revolutionary` | 삭제 |
| `I hope this email finds you well` | 삭제 (LinkedIn/email 공통 클리셰) |
| `synergy` / `leverage` / `circle back` | 삭제 |
| `I'd love to show you a demo` | `I'd love to hear your thoughts on…` |
| `solve your problem` | `learn how you approach…` |
| `increase your ROI` / `save you time` | 삭제 (세일즈 피치) |
| `schedule a demo` / `book a call` | `Would a 15-minute chat make sense?` |

## 어프로치 선택 기준

리서치 결과를 분석하여 최적의 어프로치를 선택합니다. 사용자가 명시적으로 지정할 수도 있습니다.

| 어프로치 | 최적 상황 | 톤 |
|----------|-----------|-----|
| **Discovery** (탐색형) | 상대방의 도전과제/워크플로우를 배우고 싶을 때 | "~에 대해 여쭤보고 싶어서" |
| **Design-Partner** (검증형) | 피드백/검증을 받고 싶을 때 | "맞는 방향인지 확인하고 싶어서" |
| **Peer-Exchange** (동료형) | 관련 분야 연구/경험을 공유할 수 있을 때 | "비슷한 주제를 다루고 있어서" |
| **Trigger-Based** (이벤트형) | 최근 발표/뉴스/논문이 있을 때 | "최근 ~를 보고 인상 깊어서" |
| **Warm-Intro** (소개형) | 공통 인맥/이벤트/커뮤니티가 있을 때 | "~에서 뵈었는데" |

**기본값은 Discovery** — 다른 어프로치가 더 적합한 명확한 이유가 없으면 Discovery를 사용하세요.
**동일 어프로치/제목공식 3회 이상 연속 사용 금지** — 다양하게 선택하세요.

### 사용하면 좋은 표현 (일본어)
- "~に大해 여쭤보고 싶어서 연락드립니다" → 「~についてお伺いしたくご連絡いたしました」
- "방향이 맞는지 확인하고 싶습니다" → 「方向性が合っているかご意見をいただきたく」
- "팔려는 게 아니고, 의견을 듣고 싶습니다" → 「ご意見をお聞かせいただければ幸いです」
- "대신 ~를 공유해드릴 수 있습니다" → 「代わりに~を共有させていただけます」
- "15분 정도 편하신 시간에" → 「15分ほどお時間を頂戴できましたら」
- "물론 지금 타이밍이 아니시면 전혀 괜찮습니다" → 「もちろんタイミングが合わなければご放念ください」

- 「革新的な/画期的な/業界最高の」 (과장)
- 「ROI XX%達成」 (세일즈 피치)
- 「貴社の課題を解決」 (문제를 단정짓지 말 것)
- 「限られた機会/今だけ」 (긴박감 조성)
- 「~ソリューションのご提案」 (세일즈 제목)
- 「~でお力になれます」 (도와준다는 전제 = 영업)
- 「~% 改善/削減/向上」 (성과 수치 강조)
- 「デモをお見せしたい」 (세일즈 CTA)

## 제목(Subject Line) 공식

**답장률을 결정하는 가장 중요한 요소입니다. 호기심을 유발하되 세일즈 느낌이 없어야 합니다.**

| 공식 | 패턴 | 예시 |
|---|---|---|
| **Curiosity** (호기심형) | 상대방 관심사 기반 질문 | 「CNSデータ分析で最も大きなボトルネックは？」 |
| **Their-Work** (성과언급형) | 상대방 성과/논문/발표 언급 | 「[論文/発表]に関する質問」 |
| **Shared-Interest** (공통관심형) | 공통 관심사 연결 | 「[分野]で似たアプローチをしていまして」 |
| **Event** (이벤트형) | 최근 이벤트 연계 | 「[カンファレンス/発表]後に気になった点がありまして」 |
| **Direct-Ask** (직접요청형) | 솔직한 요청 | 「15分だけお話を伺えませんか」 |

**금지 제목 (세일즈 느낌이 나는 제목):**
- 「~ソリューションのご提案」 「~をお手伝いできます」 「~% 改善」
- ROI/성과 수치 강조, 「ご紹介」 「ご提案」 등 영업 키워드

**제목 규칙:**
- 일본어: 30자 이내
- 영어: 8단어 이내 (50자 이내 권장)
- 대문자/느낌표/이모지 금지
- 스팸 트리거 단어 회피

**일본어 제목 문법 체크리스트 (필수):**

A) **「の」 탈락 금지** — 명사+ご相談 앞에 반드시 「の」를 넣는다:
| NG | OK |
|---|---|
| `効率化ご相談` | `効率化のご相談` |
| `反応性予測AIご相談` | `反応性予測AIについて` |
- 콜드메일에서는 `ご提案` 보다 `ご相談` / `について` 가 부드러움 (押し売り感 회피)

B) **「〜基盤」은 제목에서 피한다** — 논문/기술문서 톤이므로 `における` / `を用いた` 로 바꾼다:
| NG | OK |
|---|---|
| `脳波基盤反応性予測` | `脳波（EEG）反応性予測` / `EEG（脳波）を用いた反応性予測` |
| `EEG基盤の臨床反応性予測` | `EEG（脳波）を用いた臨床反応性予測` |

C) **「効率化」 앞에 동작 명사가 빠져 의미가 뭉개지면 보충한다**:
| NG (동작이 누락) | OK |
|---|---|
| `評価資料効率化` | `評価資料作成の効率化` |
| `外部候補評価効率化` | `外部候補評価の効率化` |
- 「〜効率化」 앞의 명사가 프로세스(評価, 探索)인지 산출물(資料, レポート)인지 구분:
  - 프로세스: `〜の効率化` (「の」만 추가)
  - 산출물: `〜作成の効率化` (동작 보충 + 「の」)

## CTA 전략 (부담 없는 요청)

모든 CTA는 **상대방이 부담 없이 응할 수 있는** 수준이어야 합니다.

| CTA 유형 | 전략 | 예시 (ja) |
|---|---|---|
| `chat` | 15분 짧은 대화 요청 | 「15分ほどお話を伺えませんか」 |
| `opinion` | 의견/피드백 요청 | 「ご意見をお聞かせいただけますと幸いです」 |
| `share` | 인사이트/자료 공유 제안 | 「関連の調査結果を共有させていただけますが、ご興味ありますか」 |
| `reply` | 가벼운 답장 유도 | 「ご関心があれば一言いただけますと幸いです」 |

**CTA 규칙:**
- 메일당 CTA **1개만**
- 「デモをお見せしたい」 「ご提案の時間をいただきたい」 등 세일즈 CTA 금지
- 예/아니오로 답할 수 있는 가벼운 요청
- 옵트아웃 라인 포함: 「もちろんタイミングが合わなければご放念ください」

## 이메일 구조

```
제목: [호기심을 유발하되 세일즈 느낌이 없는 제목]

{호칭},

[Hook - 1~2문장] 상대방의 구체적 연구/성과/발표를 언급하며 관심 표현

[Why-You - 1문장] 왜 이 사람에게 연락했는지 (전문성, 경험)
[Value-Exchange - 1문장] 상대방에게 줄 수 있는 것 (early access, 인사이트, 결과 공유)

{서명}
```

**길이 제한**: 제목 ≤ 30자(ja) / 50자(en), 본문 ≤ 일본어 ~200자 / 영어 100단어

## 자동 품질 체크

메일 작성 후 아래 체크리스트를 자동으로 검증합니다. 통과하지 못하면 재작성합니다:

- [ ] 상대방의 **구체적 연구/성과/발표**가 언급되어 있는가?
- [ ] **세일즈 피치 없이** 내가 하는 일을 설명하고 있는가?
- [ ] 상대방에게 **줄 수 있는 가치**가 명시되어 있는가?
- [ ] CTA가 **부담 없는** 요청인가? (15분 대화, 의견 한마디)
- [ ] 이 메일을 받으면 "이 사람은 나에게 뭔가 팔려는 거구나"라고 느낄까? → 느낀다면 재작성
- [ ] 검색에서 발견된 사실만 사용했는가? (내장 지식으로 사실 생성 금지)
- [ ] 길이 제한을 준수하는가?
- [ ] 스팸 트리거 단어가 없는가?
- [ ] "이 메일을 다른 회사에 보내도 되는가?" → YES면 개인화 부족, 재작성
- [ ] 금지 표현 리스트에 해당하는 표현이 없는가?

## 데이터 소스 (3층 구조)

콜드메일 작성 시 아래 3개 데이터를 **모두 조합**하여 활용합니다:

### 1층: CSV 컨택 리스트
CSV 파일에서 읽는 기본 정보입니다.
- `contact_name`, `company`, `contact_title`, `email`, `linkedin_url`
- 추가 컬럼이 있으면 함께 활용 (language, pain_point, our_solution 등)

### 2층: 실시간 웹 리서치 (필수)
**매번 콜드메일 작성 시 반드시 수행합니다.**
- WebSearch로 `"{회사명}" OR "{담당자명}" research OR publication OR announcement 2025 2026` 검색
- WebFetch로 회사 홈페이지, 뉴스, LinkedIn, 논문 페이지 방문
- **상대방의 구체적 성과/발표/논문을 찾는 것이 최우선** — 이것이 Hook의 재료가 됩니다
- **채용공고 확인**: `"{회사명}" careers OR jobs OR hiring` 검색 또는 회사 홈페이지의 Careers 페이지 방문
  - 우리 제품/서비스와 관련된 포지션을 채용 중이면 → Context에 자연스럽게 녹여서 "그 부분에서 도움이 될 수 있을 것 같다"는 뉘앙스 추가
  - **별도 어프로치가 아님** — Discovery 등 기존 어프로치 안에서 추가 멘트로 활용
- 수집한 정보를 CSV 기본 데이터와 조합하여 메일에 반영

### 데이터 조합 우선순위
```
2층 (실시간 리서치) > 1층 (CSV)
```
더 구체적이고 최신인 정보가 우선합니다.

## 실행 절차

1. 사용자 입력 확인 (CSV 경로, 제품 번호, 기타 지시사항 파악)
2. **`data/feedback_log.md` 읽기** — 과거 피드백을 확인하고 작성 시 반영
5. CSV 파일을 Read 도구로 읽기
6. **각 대상 회사별 실시간 웹 리서치 수행** (WebSearch + WebFetch, 회사당 최소 2회)
   - `"{회사명}" OR "{담당자명}" research OR publication OR announcement 2025 2026` 검색
   - 회사 홈페이지, 뉴스, LinkedIn, 논문 페이지 방문
   - **상대방의 구체적 성과/발표/논문을 찾는 것이 최우선**
   - 채용공고 페이지도 확인 — 우리 제품 관련 포지션 채용 중이면 Context 멘트에 활용
7. 각 대상별로:
   a. CSV + feedback_log.md + 웹 리서치 결과 데이터 조합
   b. 리서치 결과 분석 → 최적 어프로치 선택 (Discovery/Design-Partner/Peer-Exchange/Trigger-Based/Warm-Intro)
   c. 제목 공식 선택 및 작성 (**feedback_log.md의 제목 피드백 반영**)
   d. CTA 유형 결정 (부담 없는 요청)
   e. 메일 본문 작성 (Hook → Context → Why-You → Value-Exchange → CTA)
   f. 품질 체크 통과 확인 — 세일즈 피치 감지 시 재작성
8. 결과 출력 및 파일 저장
9. **전수 검수 실행 (필수)** — `/review` 스킬 기준으로 전체 메일을 검수합니다. 이 단계는 생략할 수 없습니다.
   a. 1단계: 팩트체크 (회사별 고유 사실 WebSearch 검증)
   b. 2단계: 세일즈 피치 감사 (금지 표현, 과장, 솔루션 강요 여부)
   c. 3단계: 문맥/문법 검수 (파손 패턴, 플레이스홀더, 이름/회사명 일치)
   d. 4단계: 영문 자연화 린트 (EN 메일 대상)
   e. 5단계: 일본어 자연화 린트 (JA 메일 대상)
   f. 검출된 CRITICAL / WARNING 항목은 즉시 수정 후 재검수
   g. 검수 리포트를 `output/review_YYYYMMDD.md`로 저장
10. **GMass용 HTML 변환 (필수)** — 검수 완료 후 최종 CSV의 body 컬럼에서 `\n`을 `<br>`로 일괄 변환합니다. 사용자가 받아보는 CSV는 항상 HTML 변환 완료 상태여야 합니다.
